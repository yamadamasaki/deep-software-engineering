# requirement と context の共進化

深ソフトウェア工学のこのモデル (乳化過程) では, 一度の乳化で完了することは考えていない. 乳化の過程は繰り返し行われることを前提としている.

requirement の実現は context を変化させ, context の変化は requirement の更新を促す. この相互的な応答関係を, ここでは生物学の共進化になぞらえて呼ぶ.

このひとつ一つの乳化過程は今までの以下のようなソフトウェア過程を含むソフトウェア過程に対応しており, (古典的ソフトウェア工学がフェーズ・ゲートとして切り分けるのに対して) それらを (後アジャイル・ソフトウェア工学よりもさらに徹底して) 分け隔てなく扱う.

- プロトタイピング
- スパイキング
- 開発
- デバグ
- リファクタリング
- テスト
- 機能追加
- 修正
- 保守
- 移行

これらの過程の履歴は, 一つのリポジトリの中に requirement, context, そして生成された executable のセットとして保存され, それらの変遷として記憶される.

一つの乳化過程が終了するごとに, この過程に関わるすべての行為者が集まって, その記憶を参照しつつ, 次の乳化過程をどうするかの対話が行われる.

対話は, 限られた時間の中でできるだけ早く結論を出すことを求められない. むしろ, 時間をかけても行為者全体の同意が得られることが重要である.

対話では, 以下について議論する.

- 次の乳化過程では, 何を実現するか (一つの目当てとして)
- そのために以下をどうするか
  - requirement への変更
  - context への変更
  - emulsification への変更
- トポロジ (後述) をどうするか
- このコミュニティの構成や規約, 行動に関する変更はあるか

対話自体もリポジトリに記憶される.

## 共進化のトポロジ

複数の連続する emulsification を構成する方法は, 以下のようにいくつか考えられる.

- 線形
  - 後戻りせず, 一本道で変更を積み重ねる
- 木
  - 複数の選択肢を実験して, その中から一つを選択して先に進む
- 束
  - 複数の選択肢を実験して, その中の複数の良い点悪い点を取捨選択し, 合成 (merge) して先に進む. 複数の親を持つことを許容する
- グラフ
  - 後戻りや循環を許す
- それらの組み合わせ
- その他の制約

これらのトポロジは, ソフトウェア自体の構造や, ソフトウェア開発の構造に大きな影響を与え得る. したがって, 最初からトポロジに制限を加える (例えば線形トポロジに限る, など) ことも重要になる.
