# SWEBOK v4 — 言語行為としてのソフトウェア工学: 各章の主要ポイント

> ソフトウェア工学を人と計算機との間の言語行為として見る立場から、SWEBOK v4 の各章を読み、重要なポイントを抽出する。

---

## Chapter 01: Software Requirements

**総括**: 要求とは「意図の伝達」という言語行為そのものである。

1. **二大問題は言語行為の病理** — 不完全性（言われなかった発話）と曖昧性（発語内の力の不確定）がソフトウェア要求の根本問題。要求工学の全プロセスはこの二つの克服のためにある
2. **意図 (intent) の保存と伝達** — SWEBOK は要求文書の本質を「意図の伝達」と明言。バグとは「意図されたこと」と「実際の振る舞い」のずれ——発語内行為と発語媒介行為の不一致
3. **要求開発は合意形成** — SWEBOK 自身が要求プロセスを「合意の形成と維持」と定義。要求は記述ではなく、行為者間の相互的約束行為
4. **引き出しは沈黙から発話を引き出す行為** — 暗黙知の言語化が引き出しの本質。5-whys は表面的発話の背後の真の発語内の力を掘り出す技法
5. **仕様の形式は発話の厳密さのスペクトル** — 自然言語→構造的自然言語→受入基準→形式モデル。"shall" は指示的言語行為の標識であり、テストケースとの境界は発語内の力の差異に過ぎない
6. **派生要求は言語行為のカスケード** — 同じ発話が文脈（聞き手の立場）によって宣言にも指示にもなる
7. **要求の衝突は対抗する言語行為の矛盾** — 解決は交渉（メタ言語的行為）またはプロダクトファミリ（不変/可変の分離）
8. **検証は適切性条件の確認** — 曖昧でない・検証可能・拘束力がある・真のニーズを表す——言語行為の適切性条件に対応
9. **ステークホルダー分析は発話権の制度設計** — 「人でない要求源」は認められるが「発話者」としてではない。接地先を発話者として認めるとき全プロセスが変容する
10. **Perfect Technology Filter** — 機能要求（接地先の論理に根ざす発話）と非機能要求（道具の制約に根ざす発話）を分離する枠組み
11. **暗黙知の問題は接地先の問題に通じる** — 接地先は「語れない」のではなく「語る言語を持っていない」のかもしれない

→ 詳細: [ch01-software-requirements.md](ch01-software-requirements.md)

## Chapter 02: Software Architecture

**総括**: アーキテクチャは行為者間の「共有言語の制定」である。

1. **AD は遂行的発話** — アーキテクチャ記述は事実の記述ではなく、宣言・約束・指示という遂行的言語行為
2. **ビューポイントは言語ゲーム** — SWEBOK 自身がビューポイントを「language」「vocabulary」と明記。各ビューポイントが異なる言語ゲームの規則体系を構成する
3. **ステークホルダー = 対話の参加者** — 接地先がステークホルダーに含まれていないことは「発話資格の不在」
4. **Conway の法則は対話構造の物質化** — アーキテクチャは組織の対話パターンの物質化。新たな行為者の参加はアーキテクチャの変容を必然的に要求する
5. **アーキテクチャ決定は約束的言語行為** — 技術的負債は「約束の先送り」
6. **パターン/スタイルは慣用句 (idiom)** — SWEBOK 自身が「idiom」の語を使用。繰り返し成功した言語行為の結晶
7. **ADL は言語行為の形式化** — 厳密性（利点）と表現できないものの排除（損失）の二面性
8. **評価はメタ対話** — active review は「自ら発話し応答を引き出す」行為
9. **多重ビュー問題は翻訳問題** — 異なる言語ゲーム間での一貫性の問題。言語モデルが翻訳者となりうる

→ 詳細: [ch02-software-architecture.md](ch02-software-architecture.md)

## Chapter 03: Software Design

**総括**: 設計とは「語彙の創造」による問題と解の表現である。

1. **設計思考の5ステップは言語創造の過程** — SWEBOK 自身が「linguistic nature」と明言。目的の結晶化→概念の定式化→メカニズムの考案→記法の導入→記法の使用は言語ゲームの創設に他ならない
2. **SDD は伝達の媒体** — 設計記述は行為者間コミュニケーションの媒体。暗黙の設計は対話の範囲を制約する
3. **設計原則は対話の格率** — 抽象化=関係性の格率、関心の分離=様態の格率、完全性=量の格率、カプセル化=量の格率（語りすぎるな）
4. **設計パターンは解の定型表現・語彙** — SWEBOK が「solution vocabulary」と明記。GoF の23パターンはオブジェクト指向設計の基本語彙集
5. **DSL は接地先の言葉を計算機に持ち込む試み** — 翻訳を省略しドメインの言葉で直接実装。Domain-Driven Design の「shared language」は言語の共有
6. **設計根拠はメタ言語的記録** — 「なぜそう語ったか」と「なぜそう語らなかったか」の記録
7. **倫理的設計は聴衆の社会への拡大** — 透明性と説明責任は発話者への誠実性要求の拡張

→ 詳細: [ch03-software-design.md](ch03-software-design.md)

## Chapter 04: Software Construction

**総括**: 構築とは「人間が計算機に対して行う発話行為」の実行である。

1. **構築言語は人間→計算機の発話形式の分類体系** — 設定言語（最も制約された発話）→プログラミング言語（最も自由な発話）。良いコードは「意味が直観的にわかる」発話
2. **複雑さの最小化は発話の明晰さの要請** — Grice の様態の格率のコード版。「clever code より readable code」は発話の目的が伝達にあることの宣言
3. **Design by Contract は約束的言語行為の形式化** — 事前条件=準備条件、事後条件=約束、不変条件=宣言。モジュール間の相互作用を明示的な約束の交換として形式化
4. **API は行為者間の対話プロトコル** — シグネチャ=語彙、セマンティクス=語用論。API-first=対話のルールを先に決める
5. **LLM によるコード生成は三者間対話の出現** — 人間↔LLM↔計算機。ただし人間が最終的な発話の責任者
6. **フィードバックループは構築の対話的過程化** — アジャイルとDevOpsによる継続的対話

→ 詳細: [ch04-software-construction.md](ch04-software-construction.md)

## Chapter 05: Software Testing

**総括**: テストとは「発話の検証」——意図と実行の照合行為である。

1. **テストの基本定義は発話の受理確認** — 計算機が意図通りに応答しているかの確認。無限の発話可能性から有限の確認点を選ぶサンプリング検証
2. **Oracle 問題は了解の正しさの判定問題** — Oracle=発話が正しく了解されたかを判定する審判。Dijkstra の格言は発話の意図が完全に形式化できないことの帰結
3. **テストレベルは対話の粒度の階層** — Unit=語句、Integration=語句の組合せ、System=全体の発話、Acceptance=聴き手による受理
4. **テスト技法の分類は検証の視点の分類** — Black-box=語用論的、White-box=統語論的、Experience-based=実践知
5. **ATDD は受理条件の先行宣言** — 成功条件を先に合意する理性的コミュニケーション戦略
6. **AI/ML テストは非決定的発話の検証** — 同じ入力に異なる応答をする行為者の検証という新問題
7. **ドメイン固有テストは接地先の文脈に依存した検証** — 接地先の専門家がテストに参加する必要性

→ 詳細: [ch05-software-testing.md](ch05-software-testing.md)

## Chapter 06: Software Engineering Operations

**総括**: 運用とは「発話が現実の中で生き続ける」ための営みである。

1. **Infrastructure as Code は環境の宣言的記述** — 実行環境自体を宣言的言語行為として記述。Self-documentation=自己参照的発話
2. **DevOps は開発と運用の対話統合** — 発話の生成者と管理者の分離を解消する組織的取組み。Conway の法則の逆適用
3. **監視・テレメトリは計算機の応答を聴く行為** — ログ・メトリクス・トレース=計算機の報告的発話。「hope should not be a strategy」は計算機の声を無視するなという宣言
4. **SLA はサービスの契約的言語行為** — SLI=語彙の定義、SLO=約束の内容、SLA=約束の法的形式化
5. **インシデント管理は異常報告と修復の対話プロトコル** — Post mortem=対話の失敗からの学習メカニズム
6. **ロールバックは発話の撤回の制度化** — Canary release=限定的聴衆への試行的発話

→ 詳細: [ch06-software-engineering-operations.md](ch06-software-engineering-operations.md)

## Chapter 07: Software Maintenance

**総括**: 保守とは「過去の発話を現在の文脈で生かし続ける」営みである。

1. **保守の本質は他者の発話の理解から始まる** — 発話者不在の状況での解釈。原著者のいないテクストの解釈という解釈学的状況
2. **Lehman の進化法則は発話の生態学** — Continuing Change=文脈が変われば意味が変わる。Feedback System=多層・多ループ・多行為者の対話。ソフトウェアは「終わりのない対話」
3. **保守カテゴリは改訂の動機の分類** — Corrective=言い間違いの訂正、Adaptive=環境変化への語用論的適応、Perfective=表現の洗練。80%以上がenhancement
4. **技術的負債は発話の質の劣化の蓄積** — 時間的制約下での不十分な表現の蓄積。返済=過去の不明瞭な発話を明晰に書き直すこと
5. **リバースエンジニアリングは発話からの意図の復元** — 発語内行為の再構成。保守者が考古学者のように過去の発話を発掘し解読する
6. **CI/CD は対話の速度の制度化** — 複数の行為者の発話を継続的に統合し、聴き手に素早く届ける

→ 詳細: [ch07-software-maintenance.md](ch07-software-maintenance.md)

## Chapter 08: Software Configuration Management

**総括**: 構成管理とは「発話の版と履歴の管理」の制度化である。

1. **構成項目は管理対象となる発話単位の同定** — 計画書、仕様書、コード——すべて「発話の記録」。一意識別子=発話への名前の付与
2. **ベースラインは合意された正典の確立** — 「この版を公式とする」という宣言的言語行為。聖典のカノン化に類似
3. **変更要求プロセスは改訂提案の審議プロトコル** — CCB=発話の変更権限を統制する制度的機関。無秩序な変更を防ぐ
4. **CI 間の関係は発話間の相互参照構造** — 依存・導出・継承。SBOM=発話を構成する全要素の目録
5. **構成監査は発話の整合性の検証** — FCA=意味的整合性、PCA=形式的整合性の第三者検証
6. **リリース管理は発話の公式な公開の統制** — VDD=出版物の奥付。デジタル署名=発話の真正性の保証。継続的デリバリー=出版の加速化

→ 詳細: [ch08-software-configuration-management.md](ch08-software-configuration-management.md)

## Chapter 09: Software Engineering Management

**総括**: 管理とは「対話の組織化と統制」である。

1. **ソフトウェアの可塑性は発話の可塑性に由来** — いつでも「言い直せる」ことが強みでも弱みでもある。プロジェクト失敗が social issues に帰せられる=対話の失敗
2. **要求の交渉は対話の開始と合意形成** — 「clients often do not know what is needed」=対話の前には完全な意図は存在しない
3. **見積もりは対話の見通しの形成** — 変数が個人の経験、チームの相互作用、文化——すべて対話の質に関わる要素
4. **リスク管理は対話の不確実性への対処** — 「不要な機能の追加」=聴き手が求めていないことを語る傾向——量の格率への違反
5. **監視・制御・報告はメタレベルの対話管理** — 報告は聴衆に合わせた報告的発話
6. **クロージャと回顧は対話の総括と反省** — クロージャの宣言=宣言的言語行為。回顧=対話についての対話
7. **測定は対話の効果の定量化** — 「management without measurement = lack of discipline」=対話の効果を確認せずに続ける非合理性

→ 詳細: [ch09-software-engineering-management.md](ch09-software-engineering-management.md)

## Chapter 10: Software Engineering Process

**総括**: プロセスとは「対話の手順の体系化」である。

1. **プロセス定義は対話の手順の明文化** — 入力・活動・出力=「問い→応答→成果」の形式化。プロセスの合意はメタ対話によるルール設定
2. **ライフサイクルモデルは対話の全体構造のパラダイム** — ウォーターフォール=一方向的独白、アジャイル=適応的対話、スパイラル=リスク駆動の対話
3. **アジャイルの本質はコミュニケーションの優先** — face-to-face=最も直接的な対話形式の優先。「open to change」=対話を通じて意見が変わることの受容
4. **DevOps は対話のサイロの打破** — シフトレフト=品質に関する対話を後ろから前に移動
5. **プロセス改善は対話の質の継続的向上** — PDCA=対話の改善サイクル。レトロスペクティブ=対話についての対話

→ 詳細: [ch10-software-engineering-process.md](ch10-software-engineering-process.md)

## Chapter 11: Software Engineering Models and Methods

**総括**: モデルとは「ソフトウェアについての構造化された発話」である。

1. **モデリング原則は対話の基本規範** — Grice の格率に直接対応: essentials=関係性、perspective=様態、communication=対話の根本目的
2. **構文・意味論・語用論はモデリング言語の三層構造** — SWEBOK が明示的にこの三層を記述。「同じ構文でも環境が異なれば意味が異なる」は語用論の基本的教訓
3. **事前/事後条件・不変条件は発話の約束の形式化** — 事前条件=準備条件、事後条件=約束、不変条件=前提
4. **モデル分析は発話の品質検証** — 完全性=量の格率、一貫性・正確性=質の格率、追跡可能性=発話の相互参照構造
5. **形式手法は数学的に厳密な発話** — 曖昧さを排除する試み。軽量形式手法は実用性と厳密性のバランス
6. **アジャイルメソッドは対話中心のメソッド** — 短いイテレーション=対話頻度の最大化。ペアプログラミング=同時的対話的コーディング

→ 詳細: [ch11-software-engineering-models-and-methods.md](ch11-software-engineering-models-and-methods.md)

## Chapter 12: Software Quality

**総括**: 品質とは「発話がその意図を正しく実現しているか」の総合的評価である。

1. **品質の定義は発話の成功条件の多面的記述** — 要求への適合=発語内効力の実現、用途への適合性=発語媒介効果の達成、顧客が最終審判者=了解の主権は聴き手
2. **エラー・欠陥・故障は発話の失敗の分類学** — エラー=言い間違いの原因行為、欠陥=テクストに残る痕跡、故障=通じなかった・誤解を招いた
3. **品質文化・倫理は対話の誠実性の制度的基盤** — 正確な報告=Grice の質の格率。不誠実な対話は基盤を崩す
4. **V&V は発話の二重の検証** — Verification=「言おうとしたことを正しく言えているか」、Validation=「言うべきことを言えているか」
5. **レビュー・監査は第三者による発話の検証** — 視点ベースレビュー=異なる聴き手の立場に立つ。ペアプログラミング=コーディングと検証の同時遂行
6. **ディペンダビリティは発話の信頼性への要請** — 可用性、信頼性、保守性、安全性、セキュリティ。インテグリティレベル=信頼度等級

→ 詳細: [ch12-software-quality.md](ch12-software-quality.md)

## Chapter 13: Software Security

**総括**: セキュリティとは「発話の完全性・機密性・可用性の保護」である。

1. **CIA 三要素は対話の信頼性の三条件** — 機密性=対話の秘密性、完全性=発話の正確な伝達、可用性=対話の継続可能性。真正性・否認防止=対話の帰属性
2. **セキュリティ設計は対話の防衛的設計** — 脅威モデリング=悪意ある聴き手の想定。アクセス制御=対話の参加資格の管理。暗号=対話の暗号化
3. **セキュアコーディングは脆弱性を防ぐ記述規範** — 入力検証=信頼できない話者の発話を鵜呑みにしない。最小権限=必要な範囲だけで聴く。zero trust の実現
4. **脆弱性管理は発話の欠陥の体系的追跡** — CVE=具体的な脆弱性のカタログ、CWE=脆弱性の類型学、CAPEC=攻撃パターンの分類
5. **DevSecOps はセキュリティ対話の統合** — セキュリティを「後から追加する検閲」から「対話の一部」へ
6. **ドメイン固有セキュリティは対話の文脈に応じた防衛** — ML 攻撃は「解釈の改竄」という新しい脅威

→ 詳細: [ch13-software-security.md](ch13-software-security.md)

## Chapter 14: Software Engineering Professional Practice

**総括**: 専門職実践とは「対話の倫理的・制度的基盤」の確立である。

1. **倫理規範は対話の根本的な格率** — 不適切な成果物の隠蔽=質の格率違反、情報の偽造=誠実性条件の根本的違反。倫理違反は対話の信頼基盤の毀損
2. **認証・資格・免許は対話者の能力の制度的保証** — 準備条件 (preparatory condition) の制度化。発話者が適切な権限と能力を持つことの社会的保証
3. **コミュニケーションスキルは対話の質を直接決定する能力** — 読解=解釈の能力、文書作成=発話の精密さ、プレゼンテーション=発語媒介効果の最大化
4. **チーム力学は対話の社会的条件** — 知的誠実性=質の格率、無知の承認=準備条件の誠実な表明。心理的安全性が対話の質の前提
5. **不確実性への対処は対話の不完全性との共存** — 追加的な対話による解決、または管理すべき不確定性としての受容
6. **多様性・包摂性は対話の開放性の条件** — ダークパターンの禁止=対話の誠実性の要請

→ 詳細: [ch14-software-engineering-professional-practice.md](ch14-software-engineering-professional-practice.md)

## Chapter 15: Software Engineering Economics

**総括**: 経済学とは「対話の価値判断の枠組み」である。

1. **提案と代替案は対話の方向の選択肢** — Do-nothing 代替案=対話そのものの必要性の問い直し
2. **意思決定プロセスは選択の対話の形式化** — 真の問題の理解→解の同定→基準定義→評価→選択→監視
3. **無形資産は対話の中に蓄積された暗黙知** — 方針・手続き=制度化された規範、文化・経験=暗黙の前提。SIPAC=対話の蓄積がビジネス目標にどう貢献しているかの評価
4. **見積もりは対話の見通しの推定** — 本質的に不確実。見積もりの倫理的要請=対話における誠実性の要請
5. **多属性意思決定は複数の格率の重み付け** — 「value does not always derive from money alone」は対話の多面的価値の表明
6. **生産性と手戻りは対話の効率と修復コスト** — 手戻りが最大のコスト要因=発話の質を最初から高めることが最も経済的

## 横断的知見: 全章を通じて浮かび上がるテーマ

### 1. ソフトウェア工学は本質的に対話的営みである
SWEBOK の全章を通じて、ソフトウェア工学の活動は一貫して**行為者間の対話**として記述できる。要求の引き出し、設計のコミュニケーション、コードの記述と読解、テストの実行、運用の監視、保守の理解——すべてが発話・解釈・応答の連鎖である。

### 2. Grice の対話の格率は設計原則・品質基準の統一的基盤を提供する
量の格率（必要十分に語れ）、質の格率（真実を語れ）、関係性の格率（関連することを語れ）、様態の格率（明晰に語れ）は、設計原則（ch03）、モデリング原則（ch11）、品質基準（ch12）、倫理規範（ch14）を横断する統一的な規範として機能する。

### 3. 言語行為の分類は SWEBOK の概念を体系的に照らす
- **主張的 (assertive)**: 報告、仕様記述、テスト結果——「こうである」
- **指示的 (directive)**: 要求仕様の "shall"、コーディング規約——「こうせよ」
- **約束的 (commissive)**: Design by Contract、SLA——「こうする」
- **宣言的 (declarative)**: ベースラインの確立、リリース、クロージャ——「こうなった」
- **表出的 (expressive)**: 設計根拠、レトロスペクティブ——「こう感じる・考える」

### 4. 「接地先」の不在は全章に通底する課題である
各章の「深ソフトウェア工学への問い」は、一貫して**接地先（ソフトウェアが対象とする現実世界の領域）をどのように対話に含めるか**を問うている。SWEBOK は接地先を「要求の源」としては認識するが、「対話の能動的参加者」としては位置づけていない。この不在の認識と克服が、深ソフトウェア工学の中心的課題である。

### 5. 対話の時間的構造が SWEBOK の全体を貫く
- **過去**: 設計根拠、リバースエンジニアリング、レトロスペクティブ——過去の対話の理解と学習
- **現在**: コーディング、テスト、運用——現在進行中の対話
- **未来**: 見積もり、リスク管理、計画——将来の対話の見通し
- **永続**: 保守、進化、技術的負債——対話は終わらないという認識
